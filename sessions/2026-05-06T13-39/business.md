## KPI 데이터 구조 및 API 연동 최종 점검 보고

### 1. 핵심 데이터 구조 최종 정의 (Minimum Viable Data Structure)

MVP 성공을 위해 필요한 데이터는 **'고객/거래 단위'**와 **'성과 측정 단위'**로 명확히 분리되어야 합니다. 아래는 자동화된 데이터 수집 및 보고 루틴을 위해 필요한 최소한의 데이터 필드 정의입니다.

| 데이터 그룹 | 필드명 (Field Name) | 데이터 타입 | 설명 (Description) | KPI 연관성 | 필수 여부 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **고객 정보** | `Client_ID` | String/Int | 고유 고객 식별자 (Primary Key) | 모든 KPI | 필수 |
| | `Lead_Source` | String | 리드 확보 채널 (온라인/오프라인/직접) | 전환율 분석 | 필수 |
| | `Contract_Date` | Date | 계약 또는 상담 발생일 | LTV 산정 기준 | 필수 |
| | `Service_Type` | String | 제공된 MVP 서비스 종류 (예: 리스크 분석, 컨설팅) | Upsell 분석 | 필수 |
| **재무 성과** | `Initial_Fee` | Float | 초기 서비스 비용 (MVP 상품 가격) | LTV 산정 | 필수 |
| | `Upsell_Amount` | Float | 추가 서비스 또는 상향 판매 금액 | Upsell 전환율 | 필수 |
| | `Total_Revenue` | Float | 고객 생애 총 수익 (LTV의 기초) | LTV 산정 | 필수 |
| | `Profit_Margin` | Float | 서비스 제공 후 순이익 (선택 사항) | 수익화 모델 검증 | 선택 |
| **성과 지표** | `Conversion_Rate` | Float | 리드에서 최종 계약/상담으로의 전환율 (UCR) | Upsell 전환율 | 필수 |
| | `LTV_Calculated` | Float | 고객 생애 가치 (현재까지의 누적 수익) | LTV 산정 | 필수 |
| | `Date_Recorded` | Timestamp | 데이터가 기록된 시점 | 보고 시점 관리 | 필수 |

### 2. API 연동 적합성 검토 (Data Pipeline Check)

**결론: 현재 정의된 데이터 구조는 개발팀의 API 연동에 매우 적합하며, 데이터 파이프라인 구축을 위한 최소한의 요구사항을 충족합니다.**

1.  **정규화 및 일관성:** 모든 필드는 명확한 데이터 타입(String, Float, Date)으로 정의되어 있어 데이터베이스 저장 및 통계 계산에 오류가 발생할 가능성이 낮습니다.
2.  **KPI 직접 연결:** `Upsell_Amount`, `Initial_Fee`, `Total_Revenue` 등의 재무 필드는 직접적으로 **Upsell 전환율(UCR)** 및 **고객 생애 가치(LTV)** 계산에 필요한 모든 변수를 제공합니다.
3.  **데이터 흐름 명확성:** `Lead_Source`와 `Service_Type` 필드는 마케팅 채널별 성과 분석(어떤 채널에서 가장 높은 LTV를 창출하는지)에 필수적인 교차 분석 정보를 제공합니다.
4.  **확장성:** 향후 수익화 모델이 확장되거나 새로운 서비스가 추가될 경우, `Service_Type` 필드를 확장하거나 새로운 재무 필드(예: 비용 추적)를 추가하는 방식으로 유연하게 확장 가능합니다.
